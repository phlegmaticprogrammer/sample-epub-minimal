Different Token Selection Mechanism

Compared to the original paper, we are using now a different token selection mechanism. Before, tokens had to be selected at every step. But this is only necessary for tokens that are empty. For tokens which are non-empty, we don't need to know whether to select it or not before the end of processing the current location, as this decision will only influence paths longer than the current position.

Selectors induced by orders on tokens

We usually define the selector in terms of a priority relation <<< on tokens, i.e. token uuu has a lower priority than token vvv iff u<vu < vu<v. There are then at least two ways to define a selector from this:

  • We define Sel⁡A B=A∪{u∈B∣∄v∈B.u<v}\operatorname{Sel} A\ B = A \cup \left\{u \in B \mid \nexists v \in B. u < v\right\}SelA B=A∪{u∈B∣∄v∈B.u<v}

The second definition has the advantage that it will guarantee that there are no u,v∈Tu, v \in Tu,v∈T with u<vu < vu<v. But this guarantee comes at the prize of possibly including tokens of lower priority instead of higher priority tokens that would also be available, just because the lower priority tokens were discovered first.

The first definition will never exclude high priority tokens in favour of low priority tokens, but will sometimes allow empty low-priority tokens to coexist in TTT with higher-priority tokens. It seems that this is the better definition, and more appealing to the intuitive sense of priority.

Note that if there are no empty tokens, then both definitions coincide.
